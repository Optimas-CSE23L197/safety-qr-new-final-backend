Golden rule: AI writes the skeleton, you write the business logic. Never let AI write security-critical code (JWT verification, token validation, OTP logic, payment webhook signature checks) without deep review. AI is fast at boilerplate — use it for repositories, validation schemas, test stubs, and seed data.

5. Cybersecurity Architecture

5.1 Threat model — what you're protecting

Asset Threat Impact Priority
QR scan endpoint (public) Enumeration of token_hash values, bot scraping of student emergency data Mass exposure of child medical data Critical
Parent account Account takeover via OTP replay or SIM swap, unauthorized card block Child safety risk — cards blocked, parents locked out Critical
Student emergency profile Unauthorized data modification (blood group, emergency contacts) Wrong emergency response — life risk Critical
Super admin Credential brute-force, session hijack, insider threat Full system compromise, financial fraud Critical
Payment webhooks Replayed or forged payment events → unauthorized order completion Financial loss, cards shipped without payment Critical
Token generation SSRF via QR content, token hash prediction Forged cards, unauthorized student registration High
School admin Horizontal privilege escalation (school A accesses school B data) Data breach, regulatory liability High
Encrypted PII fields Key rotation failure, plaintext logging, encryption oracle GDPR/DPDPA violation, PII exposure High

5.2 Security controls — layer by layer

Transport & Network

  • HTTPS only — HSTS header enforced
  • TLS 1.2 minimum, TLS 1.3 preferred
  • No mixed content
  • Nginx-level rate limiting as first line
  • Block private IP ranges in QR scan (SSRF)
  • Geo-block suspicious countries (optional)

Authentication

  • OTP: sha256 hash stored, never plaintext
  • OTP: 5 attempt max, hard expiry 10 min
  • OTP: rate limit 1 send per 60s per phone
  • JWT: RS256 (asymmetric) preferred over HS256
  • JWT: 15 min access, 7 day refresh
  • Refresh: rotation on every use, revoke old
  • Single-device: new login kills all old sessions

Authorization

  • RBAC: role checked at route middleware layer
  • Tenant scope: school_id from JWT, never from request body
  • Parent: can only access own students
  • School admin: can only access own school data
  • Row-level: all queries include WHERE school_id = jwt.school_id
  • Super admin routes: separate prefix + separate middleware chain

Input Validation

  • All inputs validated via Zod before controller
  • Phone: E.164 format enforced
  • Amounts: always in paise (integer), never floats
  • UUIDs: validated before DB lookup
  • File uploads: type + size checked, no executable extensions
  • SQL injection: prevented by Prisma parameterisation
  • XSS: sanitize middleware strips script tags

API Security

  • CSRF: double-submit cookie for state-changing ops
  • Rate limiting: per-IP and per-user-ID sliding window
  • HPP: prevent HTTP parameter pollution
  • Request size: 1MB body limit (50MB for file upload only)
  • Helmet: all security headers set
  • API versioning: /v1/ prefix enforced
  • Content-Type enforcement: JSON only except file upload

Data Protection

  • PII encrypted: dob, phone (emergency contacts), doctor phone
  • Encryption: AES-256-GCM, key from env (never committed)
  • No PII in logs — log student_id not student name
  • No raw errors to client — only error codes
  • Prisma: never select * — always explicit field list on sensitive models
  • Soft delete: deleted_at never truly deletes PII without explicit purge

Scan Security (Critical path)

  • token_hash: sha256 of random 32 bytes — unguessable
  • Rate limit: 10 scans per token per minute (Redis)
  • Device fingerprint: detect same device scanning many tokens
  • Honeypot tokens: fake tokens that alert if scanned
  • Trusted zones: known school IPs suppress anomaly alerts
  • INVALID result: never reveal why (expired vs revoked vs fake)

Worker & Job Security

  • BullMQ: workers run with minimal permissions
  • Job payloads: no PII — only IDs
  • Idempotency: all jobs check before re-executing
  • DLQ: failed jobs quarantined, not lost
  • Webhook signature: HMAC-SHA256 verified before processing
  • SSRF prevention: outbound HTTP through allowlist only

Monitoring & Response

  • Structured logs: every request has trace-id
  • Auth failures: logged to AuthAttempt table + Redis counter
  • Attack detection: attackLogger middleware flags patterns
  • Error alerts: Sentry for 5xx errors
  • Anomaly alerts: push + SMS to parents within 10 seconds
  • IP blocklist: auto-add after 50 failed auths in 1 hour

5.3 Security implementation checklist (Dev 3 owns this)

  • Razorpay webhook: verify x-razorpay-signature HMAC-SHA256 before ANY processing
  • Shiprocket webhook: verify shared secret from env before processing
  • OTP resend: invalidate previous OTP record before sending new one
  • Session lookup: always check is_active = true AND expires_at > now() — never just by refresh_token_hash
  • Card block: require fresh OTP even if user is already logged in
  • Card replacement: require OTP + create new RegistrationNonce + revoke old token
  • Emergency profile update: log EVERY change to ParentEditLog with old_value
  • Encrypted field decryption: only decrypt when explicitly needed (never in list queries)
  • Super admin login: rate limit to 5 attempts per IP per hour, then block
  • All /public/ scan routes: verify token_hash exists and status is ACTIVE before returning ANY data
  • Tenant scope middleware: unit test that school A cannot access school B endpoint even with valid school A JWT
  • QR generation: validate QR content is only the scan URL — no SSRF via injected URL
  • S3 presigned URLs: expiry 15 minutes max, no public-read on any bucket
  • Env vars: no secret committed to git — check with git secrets or truffleHog pre-commit hook
  • Error responses: never include stack trace, Prisma error code, or SQL snippet

5.4 Known vulnerability patterns to prevent (specific to this codebase)

IDOR on school_id — Most likely critical bug in current code If your repository functions accept school_id from the request body or params (e.g. POST /schools/:schoolId/students), verify that params.schoolId === req.user.school_id in the middleware — never trust the client. Check every repository function that takes school_id as input.
Token status bypass on scan endpoint Ensure the scan path checks status = ACTIVE. REVOKED and EXPIRED tokens must return a generic error — never return student data even to legitimate scanners. The distinction between REVOKED/EXPIRED must NOT be in the public response.
Race condition on pipeline step transitions Two concurrent requests to advance the same pipeline step must not create two OrderStepExecution rows. Use a Redis distributed lock keyed on pipeline:${order_id}:${step} with 30s TTL at the start of each step advancement.
Notification duplicate on worker retry If the notification worker crashes after sending SMS but before marking the Notification record SENT, retry will resend. The idempotency_key field added to Notification prevents the DB write, but MSG91 and Firebase have their own dedup — check their docs.

7. Pre-production Deployment Checklist

  • All environment variables set in production (verify with node src/config/env.js which should throw on missing required vars)
  • Database: connection pool size set to (CPU cores × 2) + 5, max 20
  • Redis: persistence enabled (AOF), maxmemory-policy = allkeys-lru
  • BullMQ workers: run as separate process (PM2 cluster for API, PM2 fork for each worker type)
  • PM2 ecosystem.config.js: restart on crash, max memory limit, log rotation
  • S3 bucket: no public-read ACL, only presigned URLs or CDN with signed cookies
  • Nginx: rate limiting at reverse proxy level (separate from app-level)
  • Sentry DSN configured, test error is sent and received
  • Health check endpoint GET /health returns 200 with DB ping + Redis ping + queue depth
  • Bull Board protected by super_admin session middleware
  • .env NOT in git. .env.example has all keys with dummy values
  • Production DB: run prisma migrate deploy (not prisma migrate dev)
  • Test Razorpay webhook end-to-end in staging with test event
  • Seed super admin account before go-live
  • CORS: origin whitelist set to production domain only — no wildcard
