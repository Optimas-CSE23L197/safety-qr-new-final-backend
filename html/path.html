01b Module anatomy — 5-file pattern

Every module follows the exact same 5-file pattern. When a developer opens any module folder, they always know what each file does.

*.controller.js

Parse request, call service, send response. No business logic. No DB calls. Usually 5–15 lines per endpoint.

*.service.js

Business logic. Calls repository + integrations. No direct Prisma. No req/res objects. Throwable errors only.

*.repository.js

All database queries live here. Returns plain objects. No business logic. One function per query.

*.routes.js

Express Router. Apply middleware here: auth, rbac, validate, rateLimit. Mount in routes/index.js.

*.validation.js

Zod schemas for req.body, req.query, req.params. Exported and used by validate() middleware.

*.worker.js optional

BullMQ worker for this module. Only present if the module has async jobs. Registered in workers/registry.js.

Module example — a controller calling a service:

scan/scan.controller.js 
import { scanService } from './scan.service.js';
import { success } from '../../shared/response.js';

export const processScan = asyncHandler(async (req, res) => {
  const result = await scanService.process(req.body, req.tenantId);
  success(res, result, 'Scan recorded');
});

02 Design decisions

Why each folder exists

Folder Responsibility What NOT to put here
config/ Singletons, env vars, external connections Business logic, request handling
middleware/ Cross-cutting request concerns DB queries, business decisions
modules/ All feature code Shared utilities, config, infra
shared/ Code needed by 2+ modules Module-specific logic
integrations/ Third-party SDK wrappers Business rules about when to call them
workers/ Worker process bootstrap Job logic (that lives in module *.worker.js)
jobs/ Scheduled / cron tasks Queue workers (those go in modules/)
routes/ Single router mounting all modules Route handlers, middleware definitions
webhooks/ Inbound third-party webhooks Outbound API calls

Config & env — the only rule that matters

config/env.js — validate at startup, crash early 
import { z } from 'zod';

const schema = z.object({
  NODE_ENV:     z.enum(['development', 'production', 'test']),
  DATABASE_URL: z.string().url(),
  REDIS_URL:    z.string().url(),
  JWT_SECRET:   z.string().min(32),
  // ... all env vars declared here
});

export const env = schema.parse(process.env); // throws on startup if invalid

// ✅ import { env } from '../config/env.js'
// ❌ process.env.ANYTHING directly anywhere else

Multi-tenancy — how to implement it

Every school is a tenant. The tenant ID always comes from the JWT, never from query params or request bodies — users cannot spoof their school. Apply the tenant middleware on all authenticated routes and it becomes a free filter.

middleware/tenant.js 
// After auth middleware attaches req.user:
export const tenantScope = (req, res, next) => {
  req.tenantId = req.user.schoolId; // set from JWT, always
  next();
};

// In every repository query:
export const findStudents = (tenantId, opts) =>
  prisma.student.findMany({ where: { schoolId: tenantId, ...opts } });
Critical

Super admin routes skip tenant scoping — they operate across all schools. Put them in a separate route prefix (/api/admin/) with a dedicated RBAC check, and never share repository functions between super admin and school admin paths.

Workers — same process vs separate

Start with workers in the same process. This keeps local dev simple and Railway costs low. The design is already split-ready because worker.js is a separate entry point.

workers/registry.js — central worker map 
import { ScanWorker }         from '../modules/scan/scan.worker.js';
import { TokenWorker }        from '../modules/token/token.worker.js';
import { CardWorker }         from '../modules/card/card.worker.js';
import { OrderWorker }        from '../modules/order/order.worker.js';
import { NotificationWorker } from '../modules/notification/notification.worker.js';

export const WORKERS = [
  ScanWorker, TokenWorker, CardWorker, OrderWorker, NotificationWorker
];
worker.js — separate entry point (split-ready) 
// node src/worker.js  ← runs workers only (no HTTP server)
import { WORKERS } from './workers/registry.js';

for (const Worker of WORKERS) { new Worker(); }
console.log('Workers started');
package.json scripts 
{
  "scripts": {
    "start":        "node src/server.js",            // API only
    "start:worker": "node src/worker.js",            // workers only
    "start:all":    "node src/server.js & node src/worker.js",  // local dev
    "dev":          "nodemon src/server.js"
  }
}

03 Scaling path

Every future split is just "move a folder and update an import." Nothing gets rewritten.

Phase 1 — now
Single Railway service, everything together
One process runs API + workers. Zero infra cost. start:all in Railway. Good until ~50 req/s or card generation causes latency spikes.
Phase 2 — when CPU spikes hit API latency
Split workers into a second Railway service
Add a second service in Railway. Point it at start:worker. Both connect to the same Postgres + Redis. Zero code change — just a new Railway config. Card generation and token batch jobs stop competing with HTTP requests.
Phase 2b — when public scan API needs to scale independently
Extract public API to its own service
The scan/ module has no dependencies on admin modules. Copy the module folder to a new repo or use a monorepo. Mount only scan routes. Add an Nginx or Cloudflare Worker in front. Admin API stays on the original service.
Phase 3 — when teams grow and deploys collide
Extract modules into microservices
Each module folder becomes a service. Because modules never import each other directly (they go through shared/ or publish events), this is mechanical — not architectural. Start with the highest-load module. Add an API gateway (Nginx, Traefik, or Kong) in front of all services.
Phase 4 — add payment gateway
integrations/payment/ is already the right place
The folder exists and is empty by design. Add your Razorpay or Stripe adapter there, wire it into order.service.js, and mount the inbound webhook in webhooks/. Nothing else changes.
Event-driven bridge for phase 3

When splitting into microservices, replace direct service imports with BullMQ events or Redis pub/sub. The queue names are already defined centrally — you're just changing what's on the other side of the queue from an in-process worker to a remote service.

04 5 rules for every developer

Post these in your README. New devs read these before they write their first line.

  • 01
    All DB queries go in *.repository.js — nowhere else
    If you write prisma.anything in a service or controller, it's a bug. Repositories return plain JS objects — never Prisma models. This is the rule that makes splitting into microservices cheap later.
    ✅ studentRepository.findBySchool(tenantId) ❌ prisma.student.findMany({ where: { schoolId } }) // in a service
  • 02
    Modules never import from each other — use shared/ or pass data as arguments
    If scan needs to know about a student, the scan service receives a studentId and calls the student repository directly — or the student module exposes a function in shared/. Circular imports and tight coupling are how monoliths become unmaintainable.
    ✅ import { findStudentById } from '../../shared/student.helpers.js' ❌ import { studentService } from '../student/student.service.js'
  • 03
    Never send email, SMS, or do CPU work inside a request handler
    Controllers respond fast and enqueue everything else. Notifications, card generation, token batch jobs — all go to a BullMQ queue. If something takes more than ~50ms, it belongs in a worker. This keeps p99 latency low even when Redis is slow.
    ✅ await notificationQueue.add('send-welcome', { userId }) ❌ await emailService.send(...) // blocks the request
  • 04
    Read env.js — never process.env directly
    config/env.js validates every variable at startup. The app crashes immediately with a clear message if anything is missing. If you read process.env directly in a module, missing vars surface as cryptic runtime errors at 3am instead of a clean startup failure in CI.
    ✅ import { env } from '../../config/env.js' ❌ process.env.JWT_SECRET // anywhere outside config/
  • 05
    Tenant ID always comes from the JWT — never from the request
    auth middleware sets req.user, tenant middleware sets req.tenantId from req.user.schoolId. Every repository call receives tenantId as a parameter. A user should never be able to pass a schoolId that affects what data they see. Super admin is the only exception and has separate route guards.
    ✅ studentRepository.findAll(req.tenantId, filters) ❌ studentRepository.findAll(req.body.schoolId, filters)
Bonus — file naming

Always module.type.js — e.g. scan.service.js, order.worker.js. The type suffix means you can search for "controller" across the codebase and find only controllers. Never abbreviate: ctrl.js, svc.js, repo.js are not allowed.
