Operations — The Core Step
Pre-generate all rawTokens in memory (never stored)
crypto.randomBytes(32) per token · 256-bit entropy
Hash all tokens: HMAC-SHA256(rawToken, TOKEN_HASH_SECRET)
Only hash stored in DB · rawToken shown once to admin
Atomic DB: TokenBatch + all Token rows in one transaction
All or nothing · order_id linked on every token
For each token: buildScanUrl(token.id)
UUID → base62 → HMAC sig buried at offset 8 → 28-char opaque code
Generate QR PNG per token (scan URL encoded inside)
512×512px · bulk generated · concurrency capped in qr.service
Upload all QR PNGs → S3 · path: qr/{school_id}/{token_id}.png
Pre-generate card numbers sequentially (collision-safe)
RESQID-{CODE}-{6hex} · sequential to avoid DB pool pressure
Atomic: Card row + QrAsset row per token (transaction)

⚠ Cancellation
at any stage
Super admin can cancel from any stage. Status → CANCELLED. Reason stored in OrderStatusLog.note. If payment received → status → REFUNDED after refund is processed.
PENDING → cancel (no cost)
CONFIRMED → cancel (no cost)
PAYMENT_PENDING → cancel (no cost)
ADVANCE_RECEIVED → cancel → REFUNDED
TOKEN_GENERATED → cancel → tokens REVOKED → REFUNDED
POST-PRINTING → cancel → partial refund only
SHIPPED → cannot cancel (in transit)
